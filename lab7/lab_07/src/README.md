# Часть 1. Теория: метод функциональных точек и модель COCOMO II

## 1.1 Что такое COCOMO II и чем он отличается от COCOMO

**COCOMO II** — это развитие классической модели COCOMO Боэма, адаптированное для современных реалий разработки ПО. Если исходная COCOMO (1981 г.) требовала знать размер проекта в KLOC ещё до начала работы, то COCOMO II предлагает **три модели** для разных стадий проекта:

| Модель | Когда применяется | Единица измерения |
|--------|-------------------|-------------------|
| **Модель композиции приложения** | Ранняя стадия, макетирование UI | Объектные точки |
| **Модель ранней разработки архитектуры** | До определения архитектуры | Функциональные точки / KSLOC |
| **Модель проектирования и разработки** | После фиксации архитектуры | KSLOC |

В данной лабораторной работе используются первые две модели.

---

## 1.2 Метод функциональных точек (FP)

### 1.2.1 Зачем нужен этот метод

Метод функциональных точек позволяет оценить **размер программы** через её функциональность — то, что умеет делать приложение с точки зрения пользователя, — **не зная заранее**, на каком языке и как именно оно будет написано. Затем, зная количество функциональных точек, можно перевести их в строки кода (SLOC) с учётом используемых языков программирования.

### 1.2.2 Пять функциональных типов

Вся функциональность приложения разбивается на пять категорий:

| Тип | Сокращение | Что это |
|-----|-----------|---------|
| Внешний ввод | **EI** | Транзакция, получающая данные от пользователя (кнопка, форма, ввод) |
| Внешний вывод | **EO** | Транзакция, отображающая данные пользователю (вывод таблицы, результата) |
| Внешний запрос | **EQ** | Интерактивный запрос: ввод + немедленный вывод (авторизация, поиск) |
| Внутренний логический файл | **ILF** | Данные, хранящиеся и поддерживаемые внутри системы (таблица БД, локальный файл) |
| Внешний интерфейсный файл | **EIF** | Данные из внешней системы, используемые приложением (биржевой сервер) |

### 1.2.3 Как определяется сложность каждого элемента

Для определения уровня сложности (низкий / средний / высокий) используются вспомогательные характеристики:

- **DET** — количество уникальных распознаваемых пользователем полей данных
- **RET** — количество логических групп данных (для ILF/EIF)
- **FTR** — количество типов файлов, на которые ссылается транзакция (для EI/EO/EQ)

По сочетанию этих характеристик каждый элемент получает уровень сложности, которому соответствует определённый **вес**:

| Тип | Низкий | Средний | Высокий |
|-----|--------|---------|---------|
| EI  | 3  | 4  | 6  |
| EO  | 4  | 5  | 7  |
| EQ  | 3  | 4  | 6  |
| ILF | 7  | 10 | 15 |
| EIF | 5  | 7  | 10 |

### 1.2.4 Расчёт числа функциональных точек

**Шаг 1.** Суммируем взвешенные элементы — получаем нескорректированное число FP (**UFPC**):

```
UFPC = Σ(количество × вес по каждому типу и уровню)
```

**Шаг 2.** Считаем коэффициент корректировки (**CAF**) на основе 14 характеристик продукта (каждая от 0 до 5):

```
CAF = 0.65 + 0.01 × Σ(характеристики продукта)
```

> Сумма всех 14 характеристик максимум 70, тогда CAF = 0.65 + 0.70 = 1.35.  
> При всех нулях CAF = 0.65. Диапазон CAF: от 0.65 до 1.35.

**Шаг 3.** Скорректированное число FP (**AFPC**):

```
AFPC = UFPC × CAF
```

**Шаг 4.** Перевод в строки кода с учётом языков программирования:

```
SLOC = Σ(доля языка × коэффициент строк/FP × AFPC)
```

Примеры коэффициентов: Java — 53 стр/FP, C# — 53 стр/FP, SQL — 125 стр/FP.

## 1.3 Показатель степени p

Показатель степени **p** — ключевой параметр обеих моделей COCOMO II, определяющий нелинейность зависимости трудозатрат от размера. Он рассчитывается через пять **факторов масштабируемости**:

```
p = (PREC + FLEX + RESL + TEAM + PMAT) / 100 + 1.01
```

| Фактор | Что означает | Диапазон |
|--------|-------------|---------|
| **PREC** | Новизна проекта (есть ли похожие прецеденты?) | 0 (полное отсутствие) → 6.20 (полностью аналогичные) |
| **FLEX** | Гибкость процесса разработки | 0 (жёсткая регламентация) → 5.07 (свободный процесс) |
| **RESL** | Насколько проработана архитектура и риски | 0 (все риски устранены) → 7.07 (ничего не проработано) |
| **TEAM** | Сплочённость команды | 0 (полное взаимопонимание) → 5.48 (разрозненная команда) |
| **PMAT** | Уровень зрелости процесса (шкала CMM) | 0 (уровень 5) → 7.80 (уровень 1) |

> **Важно:** у факторов RESL, TEAM, PMAT — бо́льшее значение означает бо́льшие риски и затраты (хуже). У PREC и FLEX — чем выше, тем лучше (меньше затрат).

Если p > 1.01 — проект имеет «дизэкономию масштаба» (крупный проект труднее пропорционально). Если p = 1.01 — линейная зависимость.

## 1.4 Модель композиции приложения

### Что такое объектные точки

Объектная точка — единица измерения сложности на уровне элементов интерфейса. Подсчитываются **экранные формы**, **отчёты** и **модули на 3GL** (языки третьего поколения — C#, Java и т.д.):

| Тип | Простой | Средний | Сложный |
|-----|---------|---------|---------|
| Экранная форма | 1 | 2 | 3 |
| Отчёт | 2 | 5 | 8 |
| Модуль на 3GL | — | 10 | — |

Форма считается **сложной**, если у неё много источников данных и более 30 элементов; **простой** — если один источник данных и менее 4 элементов.

### Формулы модели

**Новые объектные точки** (с учётом повторного использования кода):
```
NOP = Объектные точки × (100 − %RUSE) / 100
```

**Трудозатраты** (в человеко-месяцах):
```
PM = NOP / PROD
```
где PROD — скорость разработки (NOP/месяц). Зависит от опыта команды:

| Опыт команды | PROD (NOP/мес) |
|-------------|----------------|
| Очень низкий | 4 |
| Низкий | 7 |
| Номинальный | 13 |
| Высокий | 25 |
| Очень высокий | 50 |

**Время разработки**:
```
TM = 3 × PM^(0.33 + 0.2 × (p − 1.01))
```

## 1.5 Модель ранней разработки архитектуры

Эта модель даёт более точную оценку, учитывая дополнительные факторы трудоёмкости. Используется KSLOC, полученный из функциональных точек.

**Трудозатраты**:
```
PM = 2.45 × EArch × KLOC^p
```

где **EArch** — произведение 7 факторов трудоёмкости ранней архитектуры:

```
EArch = PERS × RCPX × RUSE × PDIF × PREX × FCIL × SCED
```

| Фактор | Название | Смысл |
|--------|---------|-------|
| **PERS** | Возможности персонала | Чем выше — тем меньше коэффициент |
| **RCPX** | Надёжность и сложность | Чем сложнее — тем выше коэффициент |
| **RUSE** | Повторное использование | Чем больше повторного использования требуется — тем выше |
| **PDIF** | Сложность платформы | Насколько сложна целевая платформа |
| **PREX** | Опыт персонала | Чем меньше опыта — тем выше коэффициент |
| **FCIL** | Инструментальные средства | Чем лучше инструменты — тем ниже коэффициент |
| **SCED** | Требуемые сроки | Очень сжатые или очень растянутые сроки — штраф |

**Время разработки** — та же формула, что и в модели композиции:
```
TM = 3 × PM^(0.33 + 0.2 × (p − 1.01))
```

# Часть 2. Как выполняется лабораторная работа (вариант 2)

## 2.1 Описание варианта

Разрабатывается **мобильное клиентское приложение брокерской системы**. Программа имеет 4 страницы:

3. **Заявки** — таблица заявок (Тип / Имя / Цена / Количество), контекстное меню (Удалить / Изменить)
4. **Новая заявка** — поля «Бумага», «Цена», «Количество», переключатель «Покупка», кнопка «Оформить»

## 2.2 Шаг 1 — Показатель степени

Сначала нужно заполнить вкладку **«Показатель степени»** и нажать «Рассчитать P». Вот как условия задачи переводятся в значения факторов:

## 2.3 Шаг 2 — Модель композиции приложения

Переходим на вкладку **«Модель композиции»**. Нужно определить экранные формы и их сложность.

### Как определить сложность формы

Сложность определяется по количеству источников данных и элементов управления на форме:
- **Простая** — 1 источник данных, менее 4 элементов
- **Средняя** — 2–3 источника или 4–20 элементов  
- **Сложная** — 3+ источника данных или более 30 элементов

### Оценка форм по страницам

**Авторизация:**
- Форма авторизации (2 поля + кнопка + чекбокс, 1 источник данных) → **средняя сложность**

**Биржевые сводки:**
- Таблица биржевых сводок (данные из двух источников: локальный список бумаг + внешний сервер цен) → **сложная**
- Диалог добавления бумаги (1 поле + 2 кнопки, 1 источник) → **средняя**

**Заявки:**
- Таблица заявок (4 колонки, 1 источник данных) → **средняя**
- Контекстное меню (2 кнопки) → **простая**

**Новая заявка:**
- Форма оформления заявки (4 поля + кнопка, 1 источник данных) → **средняя**

**Итого экранных форм:**

| Сложность | Количество | NOP-вес | Итого NOP |
|-----------|-----------|---------|-----------|
| Простые   | 1 | 1 | 1 |
| Средние   | 4 | 2 | 8 |
| Сложные   | 1 | 3 | 3 |
| **Всего** | | | **12** |

**Модули на языках 3GL** (C# и Java — оба языка третьего поколения):
- 2 модуля × 10 = **20 объектных точек**

**Итого объектных точек: 12 + 20 = 32**

**%RUSE = 0** (повторного использования не предусматривается)

```
NOP = 32 × (100 − 0) / 100 = 32
```

**Опытность команды — низкая → PROD = 7 NOP/мес**

```
PM = 32 / 7 ≈ 4.57 чел./мес.
TM = 3 × 4.57^(0.33 + 0.2 × (1.2193 − 1.01)) ≈ 5.28 мес.
```

**Численность команды:** 4.57 / 5.28 ≈ **1 работник**  
**Бюджет:** 4.57 × 90 000 = **411 300 рублей**

## 2.6 Сравнение результатов двух моделей

| Показатель | Модель композиции | Ранняя архитектура |
|------------|-------------------|-------------------|
| Трудозатраты PM | 4.57 чел./мес. | 38.11 чел./мес. |
| Время TM | 5.28 мес. | 11.62 мес. |
| Команда | ~1 человек | ~3 человека |
| Бюджет | 411 300 руб. | 3 429 900 руб. |

Расхождение — почти в **8 раз** по трудозатратам. Это нормально: модель композиции смотрит только на интерфейсную часть и даёт оптимистичный прогноз. Модель ранней архитектуры учитывает дополнительные факторы: жёсткий график (SCED = 1.43 — значительный штраф), высокую надёжность и сложность (RCPX = 1.91) и низкий опыт команды (PREX = 1.22). Именно эти три фактора дают основной вклад в увеличение EAF.
